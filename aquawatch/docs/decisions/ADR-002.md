# ADR-002: LocalStorage for Data Persistence

## Status
**Accepted** - Currently in use

## Date
2024-01-15 (Initial Development)

## Context
We needed a way to persist user data (reports, profile) without a backend server. The application runs entirely in the browser with no server-side database.

## Decision
Use browser LocalStorage for data persistence:
- `aquawatch_reports_bengaluru` - Stores the reports array
- `aquawatch_user_profile` - Stores the current user object

## Alternatives Considered

### Option 1: IndexedDB
- **Pros**: More storage capacity, better performance for large datasets
- **Cons**: More complex API, overkill for current needs
- **Rejected**: LocalStorage is sufficient for prototype

### Option 2: SessionStorage
- **Pros**: Simpler API
- **Cons**: Data lost when browser closes
- **Rejected**: We need persistent data

### Option 3: Cookies
- **Pros**: Can be sent to server
- **Cons**: 4KB limit, not suitable for images or large data
- **Rejected**: Too small

### Option 4: No Persistence
- **Pros**: Simpler code
- **Cons**: Data lost on refresh
- **Rejected**: User experience needed persistence

## Implementation Details

```javascript
// On load - retrieve data
const reports = JSON.parse(localStorage.getItem('aquawatch_reports_bengaluru')) || initialReports;

// On change - save data
localStorage.setItem('aquawatch_reports_bengaluru', JSON.stringify(reports));
```

## Consequences

### Positive
- ✅ Zero configuration required
- ✅ Works offline
- ✅ Simple API (get/set JSON)
- ✅ Sufficient for prototype scale

### Negative
- ❌ Limited storage (~5-10MB)
- ❌ Browser-specific (can't share data between browsers)
- ❌ Can be cleared by user
- ❌ No server backup
- ❌ Base64 images fill storage quickly

## Storage Limits by Data Type

| Data Type | Estimated Size | Storage Impact |
|-----------|---------------|----------------|
| Report (text only) | ~500 bytes | ~2,000 reports per MB |
| Report with photo | ~50-500KB | 2-20 photos per MB |
| User profile | ~1KB | Negligible |

## Future Considerations

For production, consider:
- **Firebase**: Cloud sync, authentication, real-time
- **Supabase**: PostgreSQL-based, open source
- **AWS Amplify**: Full AWS ecosystem
- **IndexedDB**: If storing many photos
- **Cloud Storage**: For image uploads (AWS S3, Cloudinary)
