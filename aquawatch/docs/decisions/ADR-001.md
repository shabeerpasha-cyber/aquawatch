# ADR-001: Single HTML File Implementation

## Status
**Accepted** - Currently in use

## Date
2024-01-15 (Initial Development)

## Context
We needed a simple, portable web application for reporting WASH issues in Bengaluru. The project started as a quick prototype/demo that needed to be:
- Easily shareable (single file)
- Runnable without a build step
- Quick to iterate on

## Decision
Use a single HTML file with:
- Embedded React 18 (via CDN)
- Embedded Babel for JSX compilation
- All components, styles, and logic in one file

## Alternatives Considered

### Option 1: Create React App (CRA)
- **Pros**: Industry standard, good tooling
- **Cons**: Requires Node.js, build step, complex setup
- **Rejected**: Too complex for prototype

### Option 2: Vite + React
- **Pros**: Faster, modern tooling
- **Cons**: Still requires build step and Node.js
- **Rejected**: Overkill for single-file requirement

### Option 3: Vanilla JavaScript
- **Pros**: No dependencies, pure HTML/JS
- **Cons**: Harder to manage state, no component reusability
- **Rejected**: Less maintainable for complex UI

### Option 4: Vue.js CDN
- **Pros**: Simpler API, good for prototypes
- **Cons**: Team had React experience
- **Rejected**: Standardize on React

## Consequences

### Positive
- ✅ Single file can be shared via email/chat
- ✅ No build server needed
- ✅ Instant iteration - edit and refresh
- ✅ Easy to understand - all code in one place
- ✅ Zero configuration

### Negative
- ❌ No hot module replacement (HMR)
- ❌ Slower initial load (CDN + Babel)
- ❌ Harder to maintain as app grows
- ❌ No tree-shaking for unused code
- ❌ Larger file size

## Future Considerations

When the project requires:
- **Complex features**: Consider migrating to Vite + React build
- **Team collaboration**: Add proper linting and testing
- **Production deployment**: Optimize bundle size with build step

---

*This ADR documents why we chose single-file architecture for the prototype. Future ADRs should evaluate migration if complexity increases significantly.*
